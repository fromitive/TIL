
> indent(인덴트, 들여쓰기) depth를 2를 넘지 않도록 구현한다. 1까지만 허용한다.
    - 예를 들어 while문 안에 if문이 있으면 들여쓰기는 2이다.
    - 힌트: indent(인덴트, 들여쓰기) depth를 줄이는 좋은 방법은 함수(또는 메서드)를 분리하면 된다.

들여쓰기가 깊어지면 코드가 복잡해지고 이해하기 어려운 코드가 작성된다고 생각한다.

> 3항 연산자를 쓰지 않는다.
   else 예약어를 쓰지 않는다.
    - else 예약어를 쓰지 말라고 하니 switch/case로 구현하는 경우가 있는데 switch/case도 허용하지 않는다.
    - 힌트: if문에서 값을 반환하는 방식으로 구현하면 else 예약어를 사용하지 않아도 된다.

3항 연산자 및 else를 허용하게 되면 다양한 분기가 발생하게 되기 때문에 이해하기 어려운 코드가 작성된다.  else를 허용하지 않는 원칙은나를 위한 원칙 보다는 통제되지 않는 동료들의 코드를 방지하기 위한 원칙으로 들린다.

내 코드를 보고 else를 사용하고 있다면 동료들을 마음껏 else를 사용할 것이고 그 결과 복잡한 코드가 작성될 가능성이 있기에 처음부터 else와 3항 연산자를 사용하지 않고 코드를 작성한다면, 동료 개발자가 else를 작성할 때 코드베이스에 더욱 두드러져 보일 것 같다.

> 모든 기능을 TDD로 구현해 단위 테스트가 존재해야 한다. 단, UI(System.out, System.in) 로직은 제외
    - 핵심 로직을 구현하는 코드와 UI를 담당하는 로직을 구분한다.
    - UI 로직을 InputView, ResultView와 같은 클래스를 추가해 분리한다.

`질문`: 자신이 설계한 도메인이 제대로 동작하는 지 테스트를 한다. 그러나, 유의미한 테스트를 작성해야 할 것이다. getter를 굳이 테스트를 해야 하나?

해결하고자 하는 도메인에 따라 UI로직이 달라진다. 좋은 코드를 작성하기 위해선 UI와 관련된 DTO같은 객체에 집중하는 것이 아니라 각 객체와의 관계를 잘 설계하는 것이 중요하다고 생각한다. domain이 바뀌면 UI와 관련된 객체가 전부 바뀌기 때문이다.

> 함수(또는 메서드)의 길이가 10라인을 넘어가지 않도록 구현한다.
    - 함수(또는 메소드)가 한 가지 일만 하도록 최대한 작게 만들어라.

`질문`: 다양한 객체안에 10라인이 넘지 않는 메서드 Vs 하나의 객체의 10라인이 넘는 메서드

10라인을 넘기지 않고 구현하는건 인정한다. 로직이 단순해지기 때문이다. 그러나, 무분별하게 객체를 늘리는 행위는 코드의 복잡성을 증가시키는 행위라고 생각한다. 10 라인에 집중하지 말고. 객체간의 관계를 잘 설계하는 것이 더 중요하지 않을까?

> 배열 대신 컬렉션을 사용한다

컬렉션은 Stream API를 제공한다. 아이템이 있거나, 없거나 늘 동일한 기능을 보장해준다. 배열은 배열의 크기를 알아야 하고, index를 계산해야 하는 불편함이 잇다.

> 모든 원시 값과 문자열을 포장한다

원시값을 검증할 수 있고 책임을 분산시킬 수 있어 객체가 가벼워진다. 테코블의 내용 중 `사람`안에 `이름`과 `나이`를 기억하자. 사람 안에서 원시값인 `이름`과 `나이`를 검증하게 되면 객체가 무거워지기 때문에 중요한 기능에 집중하기 어렵다. 원시값을 포장함으로써, `사람`은 더이상 `이름`이나 `나이`를 검증하지 않고, 로직에 집중할 수 있게 된다.

> 줄여 쓰지 않는다(축약 금지).

메서드의 네이밍으로 어떤 동작을 하는지 표현하자.  줄여쓰게 되면 나중에 코드를 볼 때 무슨 의도로 작성 됐는지 추측해야 하고. 정신적 비용이 든다.

모든 엔티티를 작게 유지하면, 변경에 용이하다고 생각한다.

> 게터/세터/프로퍼티를 쓰지 않는다 

`질문`: 세터는 오케이 근데 게터는? 

`답변`: Getter를 사용하지 말라는 뜻은 모든 Getter 사용하지 말라는 뜻이 아니다.
 Getter로 객체의 필드값을 노출 시켜 해당 필드를 직접 수정할 때, getter를 호출한 객체의 상태가 변경되면 안되는다는 말이다.
